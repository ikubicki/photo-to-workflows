# Specification: Replace extractJson with BAML

## Goal

Replace manual JSON extraction and validation (`extractJson()`, `isValidWorkflow()`) in the reasoning model pipeline with BAML's schema-aligned parsing, providing typed `Workflow` output directly from the LLM call while preserving the existing API contract and token usage tracking.

## User Stories

- As a developer, I want the reasoning model's output to be automatically parsed into a typed `Workflow` object so that I can eliminate fragile regex-based JSON extraction code.
- As a user, I want the same `{ workflow, usage }` API response so that the frontend continues to work without changes.
- As a developer, I want parse failures to still return raw LLM text so that I can debug model output issues.

## Core Requirements

1. Create BAML schema files (`backend/baml_src/`) defining `Workflow`, `Stage`, `Participant`, `Decision`, `StageDependency` types matching `fixtures/types.ts`
2. Define an `openai-generic` BAML client pointing to LM Studio with `env.LM_STUDIO_URL` for the base URL and model `openai/gpt-oss-20b` at temperature 0.5
3. Define an `ExtractWorkflow` BAML function accepting `imageDescription`, `contacts`, and `instructions` string parameters, returning `Workflow`, with a prompt template using `{{ ctx.output_format }}`
4. Configure BAML TypeScript ESM generator outputting to `backend/baml_client/`
5. Replace `buildWorkflow()` internals with a call to `b.ExtractWorkflow()`, passing contacts JSON and business rules as runtime parameters
6. Simplify `analyzeWorkflow()` by removing `loadWorkflowTypes()`, `extractJson()`, `isValidWorkflow()`, and the `JSON.parse` + fallback block
7. Track token usage from the BAML call via Collector API, mapping `input_tokens`/`output_tokens` to existing `promptTokens`/`completionTokens` format
8. Catch `BamlValidationError` and return `{ raw: rawText, usage }` to preserve the current fallback behavior
9. Add `@boundaryml/baml` dependency and `baml:generate` script to `backend/package.json`
10. Add `baml_client/` to `.gitignore`

## Technical Approach

### BAML Schema Files

All files go in `backend/baml_src/`.

**`types.baml`** ã Type definitions mirroring `fixtures/types.ts`:

```baml
enum Decision {
  approved
  rejected
  change_requested @description("When changes to the submission are needed")
  pending
  completed
}

class Participant {
  name string @description("Display name of the participant")
  id string? @description("UUID of matched contact ã omit if no match found")
  role "approver" | "reviewer" | "readonly" @description("Defaults to approver if unclear from diagram")
  decision Decision? @description("Participant's decision, defaults to pending")
}

class StageDependency {
  parentStageId string @description("ID/name of the parent stage this depends on")
  condition "decision" | "deadline" | "completion" @description("Type of dependency trigger")
  decision Decision? @description("Required when condition is decision")
  deadline string? @description("ISO 8601 date string, required when condition is deadline")
}

class Stage {
  name string @description("Name of the approval stage")
  participants Participant[] @description("People involved in this stage")
  dependsOn StageDependency[]? @description("Dependencies on other stages")
  deadline string? @description("ISO 8601 deadline for the stage")
  decision Decision? @description("Overall stage decision, defaults to pending")
  metadata map<string, string>? @description("Optional key-value metadata")
}

class Workflow {
  name string @description("Name of the approval workflow")
  stages Stage[] @description("Ordered list of workflow stages")
  metadata map<string, string>? @description("Optional key-value metadata")
  decision Decision? @description("Overall workflow decision, defaults to pending")
}
```

Type mapping notes:
- `Decision` enum: BAML enums are string-valued by default ã no `= 'approved'` needed
- `deadline?: Date` ã `deadline string?` ã BAML has no datetime type; use ISO 8601 strings
- `metadata?: Record<string, any>` ã `metadata map<string, string>?` ã BAML has no `any` type
- `decision?: Decision | Decision[]` on Workflow ã `decision Decision?` ã simplified; union adds parsing complexity with minimal benefit for this use case
- `role: 'approver' | 'reviewer' | 'readonly'` ã literal union syntax (supported since BAML v0.61.0)

**`clients.baml`** ã LM Studio client:

```baml
client<llm> ReasoningModel {
  provider "openai-generic"
  options {
    base_url env.LM_STUDIO_URL
    model "openai/gpt-oss-20b"
    temperature 0.5
  }
}
```

Notes:
- `env.LM_STUDIO_URL` reads the `LM_STUDIO_URL` environment variable at runtime
- `openai-generic` provider sends no auth header by default (matches LM Studio's no-auth requirement)
- No retry policy ã matches current `maxRetries: 0` behavior
- Vision model client is NOT defined here ã it stays on OpenAI SDK

**`functions.baml`** ã ExtractWorkflow function:

```baml
function ExtractWorkflow(imageDescription: string, contacts: string, instructions: string) -> Workflow {
  client ReasoningModel
  prompt #"
    {{ _.role("system") }}
    {{ instructions }}

    Contacts list:
    {{ contacts }}

    {{ ctx.output_format }}

    {{ _.role("user") }}
    Here is the workflow diagram description:
    {{ imageDescription }}
  "#
}
```

Notes:
- `instructions` parameter receives the business rules (contact matching, OCR guidelines, role defaults) as a runtime string ã NOT hardcoded in the template
- `{{ ctx.output_format }}` auto-generates JSON format instructions from the `Workflow` type schema ã replaces the current manual TypeScript type embedding
- `{{ _.role("system") }}` and `{{ _.role("user") }}` set message roles
- The function returns `Workflow` directly ã BAML's SAP parser handles all JSON extraction, fence stripping, and type validation

**`generators.baml`** ã TypeScript ESM generator:

```baml
generator target {
  output_type "typescript"
  output_dir "../"
  module_format "esm"
  version "0.85.0"
}
```

Notes:
- `output_dir "../"` outputs `baml_client/` relative to `baml_src/` ã lands in `backend/baml_client/`
- `module_format "esm"` required ã backend uses `"type": "module"` in `package.json`
- `version` should match the installed `@boundaryml/baml` package version

#### Modify: `buildWorkflow()` ã returns business rules string

The current `buildWorkflow()` function (lines 105ã147) constructs a system prompt with embedded types and contacts, calls OpenAI SDK, and returns raw text. This is replaced by:

1. Extract the business rules from the current system prompt (lines 109ã135) into a standalone function `buildInstructions(contactsJson: string): string` that returns the instruction text WITHOUT type definitions or JSON format directives (those come from BAML's `{{ ctx.output_format }}`).

2. The extracted instructions include: contact matching rules, OCR fuzzy matching, role defaults, "IMPORTANT RULES" section.

The old `buildWorkflow()` function is deleted entirely.

#### Modify: `analyzeWorkflow()` flow

Current flow (lines 149ã177):
```
interpretImage ã loadContacts + loadWorkflowTypes ã buildWorkflow ã extractJson ã JSON.parse ã isValidWorkflow
```

New flow:
```
interpretImage ã loadContacts ã b.ExtractWorkflow (with Collector) ã map usage ã return
```

The modified `analyzeWorkflow()` will:

1. Keep Step 1 (`interpretImage`) unchanged
2. Keep `loadContacts()` call (remove `loadWorkflowTypes()`)
3. Create a `Collector` instance for token tracking
4. Build the instructions string via `buildInstructions()`
5. Call `b.ExtractWorkflow(imageDescription, contactsJson, instructions, { collector })` ã this replaces `buildWorkflow()` + `extractJson()` + `JSON.parse()` + `isValidWorkflow()`
6. Map Collector token data to `TokenUsage`:
   - `collector.last.usage.input_tokens` ã `usage.promptTokens`
   - `collector.last.usage.output_tokens` ã `usage.completionTokens`
   - Sum ã `usage.totalTokens`
7. Return `{ workflow, usage }`
8. Catch `BamlValidationError`: extract `raw_output`, return `{ raw: e.raw_output, usage }`

### Error Handling

BAML call wrapped in try/catch:
- **Success path**: `b.ExtractWorkflow()` returns a typed `Workflow` object ã return `{ workflow, usage }`
- **BamlValidationError**: SAP parser failed to produce a valid `Workflow` ã extract `e.raw_output` (the raw LLM text), return `{ raw: e.raw_output, usage }` ã preserves current frontend fallback behavior
- **Other errors** (network, LM Studio down): propagate as-is ã Fastify's error handler returns 500

### Token Usage Tracking

Two separate token tracking paths, accumulated into the same `TokenUsage` object:

1. **Vision model** (unchanged): `addUsage(usage, response.usage)` from OpenAI SDK response
2. **Reasoning model** (new): BAML Collector API after `b.ExtractWorkflow()` completes:
   ```
   collector.last.usage.input_tokens ã usage.promptTokens += ...
   collector.last.usage.output_tokens ã usage.completionTokens += ...
   (sum) ã usage.totalTokens += ...
   ```

The Collector is passed to the BAML call via `b.ExtractWorkflow(imageDescription, contacts, instructions, { collector })`.
